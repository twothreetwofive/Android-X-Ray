# 동적 분석 시나리오 정의서 (D)

`src/dynamic_analyzer/scenarios.py`에 정의된 시나리오를 사람이 읽을 수 있게
정리한 문서. 코드와 항상 같이 맞춰야 하며, 새 시나리오를 추가하면 여기도
같이 갱신한다.

## 왜 시나리오 자동화가 필요한가

A의 `frida_controller.py`는 앱을 spawn하고 후킹을 붙이고 `resume()`으로
실행을 재개하는 것까지만 한다. 하지만 대부분의 악성 행위(자격증명 탈취,
권한 오남용)는 사용자가 화면을 조작해야 실제로 코드 경로가 실행된다 —
앱을 그냥 띄워놓기만 하면 로그인 시 호출되는 `Cipher.doFinal` 같은 후킹은
한 번도 안 잡힌다. 그래서 D는 adb로 화면 조작을 자동 재현해서, B의
`hooks.js`가 실제로 의미 있는 순간에 데이터를 잡을 수 있게 만든다.

## 시나리오 목록

### 1. `launch_only` (베이스라인)

- **대상**: `com.google.android.calendar` (A/C가 지금까지 검증에 쓴 앱, 좌표 불필요)
- **목적**: 아무 조작 없이 앱을 띄웠을 때 초기 구동 과정에서 어떤 후킹이
  잡히는지 기준선을 만든다. 이후 로그인/권한 시나리오와 비교해서 "조작으로
  인해 추가로 발생한 이벤트"를 구분하는 데 쓴다.
- **스텝**: 3초 대기 후 관찰 종료.
- **현재 상태**: 좌표가 필요 없어서 지금 바로 실행 가능. 재현성 확인
  (`run_scenario_repeated`)도 이 시나리오로 먼저 검증한다.

### 2. `login_flow`

- **대상**: `__TARGET_PACKAGE__` (실제 분석 대상 앱 미정 — 자리만 만들어둠)
- **목적**: 아이디/비밀번호 입력 후 로그인 버튼을 누른다. Anubis류 뱅킹
  트로이목마가 자격증명을 암호화해서 전송하는 지점이라, `cipher`/`base64`
  후킹이 실제로 유의미하게 잡힐 가능성이 가장 높은 시나리오.
- **스텝**: 아이디 입력란 탭 → 텍스트 입력 → 비밀번호 입력란 탭 → 텍스트
  입력 → 로그인 버튼 탭 → 3초 관찰.
- **미완료**: 탭 좌표 3곳이 전부 `(0, 0)` 플레이스홀더다. 로그인 화면이
  있는 실제 대상 앱(anubis.apk 등)이 이 저장소에 없어서, 좌표는 실제 앱을
  jadx로 열어 화면 리소스 ID 확인 후 채워야 한다 (B의 `custom_xor` 미완료와
  같은 이유 — 실 샘플 부재).

### 3. `permission_request`

- **대상**: `__TARGET_PACKAGE__` (실제 분석 대상 앱 미정)
- **목적**: 런타임 권한(예: SMS 읽기)을 요청/허용해서 권한 오남용 패턴을
  재현한다.
- **스텝**: 2초 대기 → `pm grant`로 권한 부여 → 3초 관찰.
- **설계 결정**: 권한 다이얼로그를 좌표로 탭하는 대신 `adb shell pm grant`를
  기본 경로로 썼다. 다이얼로그 버튼 좌표는 Android 버전/기기 해상도마다
  달라져서 재현성이 떨어지는 반면, `pm grant`는 항상 결정적으로 동작한다.
  다이얼로그가 실제로 뜨는 것 자체를 관찰해야 하면
  `AdbRunner.tap_permission_dialog()`로 대체 가능 (좌표는 화면 비율 근사치,
  실기기 검증 필요).

## 엣지 케이스 대응 (4일차 항목)

| 상황 | 대응 |
|---|---|
| 디바이스 미연결 | `_precheck()`에서 `adb devices` 확인 후 즉시 실패 처리, spawn 자체를 시도하지 않음 |
| 앱 미설치 | `_precheck()`에서 `pm list packages`로 확인 후 즉시 실패 처리 |
| spawn 직후 크래시 | `resume()` 후 `wait_for_process()`로 최대 10초 폴링, 못 뜨면 `crashed=True` |
| 시나리오 도중 크래시 | 스텝 사이마다 `is_running()` 확인, 죽으면 남은 스텝 건너뛰고 즉시 정리 |
| 재현성 낮음 (실행마다 이벤트 수 들쭉날쭉) | `run_scenario_repeated()`로 N회 반복 실행 후 `event_counts`의 최소/최대값 비교 — C의 4주차 과제에서 나온 `waitForJava` 레이스 컨디션처럼 타이밍 문제가 있으면 편차로 드러남 |
| 리포트 덮어쓰기 (A/B가 4주차 보고서에서 지적한 미해결 항목) | 시나리오명 + 실행 시각을 파일명에 포함 (`dynamic_report_{시나리오}_{시각}.json`)해서 실행마다 별도 파일로 남김 |

## 아직 안 된 것 (5일차 팀 통합 때 같이 확인 필요)

- `login_flow`/`permission_request`의 실제 좌표·권한 문자열은 anubis.apk(또는
  대체 분석 대상)가 준비되면 채워야 함.
- 이 환경(개발 컨테이너)엔 adb/frida-server/에뮬레이터가 없어서
  `scenario_runner.py`를 실제로 실행해서 검증하지 못했음 — 문법 검사
  (`python -m py_compile`)만 통과한 상태. 팀 에뮬레이터가 있는 환경에서
  실행 확인 필요.

# 동적 분석 모듈 개발 역할 분배

### 추천 스케줄 (5일 기준, 전원 동일 기간)

**"A/B가 최소 동작 버전을 언제까지 넘겨야 C/D가 안 밀리는지"** 기준으로 짜볼게.

### 핵심 원칙: 3일차를 "인수인계 데드라인"으로 고정

C는 A(세션)+B(후킹) 둘 다 있어야 실제 메시지 흐름을 검증할 수 있고, D는 A(제어 스크립트)가 있어야 실제 앱 자동 구동을 검증할 수 있어. 그러니 **A, B는 3일차까지 "완벽하진 않아도 실제로 동작하는 최소 버전"을 반드시 넘겨야** C/D가 4~5일차에 제대로 통합/검증할 시간이 생겨.

### 5일 스케줄 (A-예원 , B-소정 , C-은아 , D-은서 ) 비중 a > c > b > d

#### 1일차 — 각자 독립 가능한 부분부터

- **A**: frida-server 연결 확인, 세션 생성(attach/spawn) 로직 작성 시작
- **B**: 대상 앱 클래스/함수 구조 분석, 후킹 대상 함수 목록 확정 (`StringBuilder.append`, `Base64`, `Cipher.doFinal` 등)
- **C**: 메시지 JSON 스키마 설계 (A/B 필요 없음) — 확정되는 대로 **A, B한테 공유** ("JS에서 이 필드명으로 `send()` 해줘")
- **D**: 테스트 시나리오 목록 작성(로그인, 권한 요청 등), `adb shell` 명령으로 앱 수동 실행 자동화 초안 (A 없이도 기본 adb 명령으로 프로토타입 가능)

#### 2일차 — 각자 핵심 로직 완성

- **A**: attach/spawn 로직 완성, `resume()` 흐름까지 동작 확인
- **B**: `Interceptor.attach`로 실제 후킹 코드 작성, **frida CLI로 단독 테스트** (`frida -U -f 패키지명 -l hooks.js`) — Python 없이도 후킹 자체는 검증 가능
- **C**: C가 만든 스키마 기준으로 메시지 파서 구조 설계 (아직 실데이터 없이 더미로)
- **D**: `scenario_runner.py` 뼈대 작성, adb 자동화 로직 보강

#### 3일차 — ⭐ 인수인계 데드라인 ⭐

- **A**: **`frida_controller.py` 최소 동작 버전 D한테 전달** (spawn→resume 되는 상태)
- **B**: **`hooks.js` 최소 동작 버전 C한테 전달** (문자열/Base64/Cipher 후킹 되는 상태)
- **C**: A의 세션 + B의 후킹 스크립트를 실제로 연결해서 `script.on('message', ...)`로 **첫 실데이터 수신 테스트**
- **D**: A의 제어 스크립트로 실제 시나리오(로그인 등) 자동 실행 테스트 시작

#### 4일차 — 각자 마무리 + 검증

- **A**: 재시도/타임아웃 처리, 배치 실행(여러 앱) 지원 추가
- **B**: 커스텀 XOR 등 표준 API로 못 잡는 패턴 보강, 후킹 노이즈 필터링 조언 (C한테)
- **C**: 실제 후킹 데이터 기반으로 필터링(중복 제거, 평문 판별) 로직 완성, `dynamic_report.json` 1차 출력
- **D**: 시나리오 반복 실행 재현성 확인, 엣지 케이스(앱 크래시, 미실행) 대응 로직 추가

#### 5일차 — 전원 통합

- 전원이 실제 앱(정상 1개 + 악성 1개)으로 A→B→C→D 전체 파이프라인 end-to-end 실행
- 버그 수정, `dynamic_report.json` 최종 형식 확인
- 발표자료(데모 캡처, 실행 로그) 준비

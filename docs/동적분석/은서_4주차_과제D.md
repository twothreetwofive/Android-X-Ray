# 4주차 과제_D

**사람**: 은서 (본인)

역할 분배 원안(`동적 분석 모듈 개발 역할 분배.txt`) 기준 D 역할 — 실행 시나리오
자동화(ADB/에뮬레이터). 산출물: `scenario_runner.py`, 시나리오 정의서.

코드는 `feature/dynamic-agent` 브랜치 `src/dynamic_analyzer/`에 있음:
`adb_runner.py`, `scenarios.py`, `scenario_runner.py`.

---

## 1일차 — 테스트 시나리오 목록 작성

A/B/C의 3일차 인수인계 전이라 A(제어 스크립트) 없이도 만들 수 있는 부분부터
시작. `scenarios.py`에 시나리오 3개를 정의:

1. **`launch_only`** — 조작 없이 앱만 띄우고 관찰하는 베이스라인. 좌표가
   필요 없어서 어떤 앱에도 바로 적용 가능. A/C가 지금까지 검증에 쓴
   `com.google.android.calendar`를 기본 대상으로 씀.
2. **`login_flow`** — 아이디/비밀번호 입력 후 로그인. 뱅킹 트로이목마류가
   자격증명을 암호화해서 전송하는 지점이라 `cipher`/`base64` 후킹이 실제로
   잡힐 가능성이 가장 높은 시나리오.
3. **`permission_request`** — 런타임 권한 요청/허용. 권한 오남용 패턴 재현용.

상세 설명은 `docs/동적분석/시나리오_정의서.md` 참고.

adb 수동 실행 자동화 초안으로 `adb_runner.py`도 이 시점에 같이 시작:
`check_device()`(디바이스 연결 확인), `is_installed()`/`is_running()`(앱
상태 확인), `launch()`(monkey 런처 인텐트로 메인 액티비티 이름 몰라도 실행).

## 2일차 — scenario_runner.py 뼈대 + adb 자동화 보강

`adb_runner.py`에 화면 조작 프리미티브 추가: `tap`/`swipe`/`input_text`/
`press_key`. 권한 요청은 다이얼로그 좌표를 탭하는 대신 `adb shell pm grant`로
처리하도록 결정 — 좌표는 Android 버전/기기 해상도마다 달라져서 재현성이
떨어지는 반면 `pm grant`는 항상 결정적으로 동작함. 좌표 탭 방식도
`tap_permission_dialog()`로 남겨둠 (다이얼로그 자체를 관찰해야 할 때 대비).

`scenario_runner.py`는 이 시점엔 A의 `frida_controller.py`가 아직 없어서
뼈대만: `ScenarioRunner` 클래스 구조, 시나리오 스텝 실행기(`_execute_step`)까지.

## 3일차 — ⭐ A의 제어 스크립트로 실제 시나리오 자동 실행 테스트 시작

A가 `frida_controller.py` 최소 동작 버전을 전달(`feature/dynamic-agent`
브랜치에 push, `FridaController` 클래스: `connect() → spawn_and_attach() →
load_script() → resume()`). 이를 받아서 `scenario_runner.py`의
`run_scenario()`를 완성:

```
_precheck()                              # 디바이스 연결 + 앱 설치 여부 확인
controller.spawn_and_attach(package)
controller.load_script(hooks_js_path)
controller.script.on("message", message_parser.on_message)  # C의 리스너 추가 등록
controller.resume()
adb.wait_for_process(...)                # 크래시 없이 실제로 떴는지 확인
시나리오 스텝 실행 (adb)
message_parser.build_report(...)         # C의 파서로 리포트 생성
```

`controller._on_message`는 A가 만든 콘솔 출력용 리스너라 그대로 두고, C의
`message_parser.on_message`를 별도 리스너로 추가 등록하는 방식을 택함 — A의
코드를 건드리지 않고 C의 데이터 수집 로직을 얹을 수 있음 (frida의
`Script.on()`은 같은 이벤트에 리스너를 여러 개 등록할 수 있음).

## 4일차 — 시나리오 반복 실행 재현성 확인 + 엣지 케이스 대응

**재현성**: `run_scenario_repeated()` 추가 — 같은 시나리오를 N회 반복 실행해서
매번 `force_stop()`으로 초기화 후 `total_events_filtered` 개수를 비교.
편차가 크면 타이밍 문제(C가 4주차 과제에서 겪은 `waitForJava` 레이스
컨디션 같은 것)가 있다는 신호로 판단.

**엣지 케이스**:
- 디바이스 미연결 / 앱 미설치 → `_precheck()`에서 spawn 시도 전에 걸러서
  실패 사유를 바로 반환.
- spawn 직후 크래시 → `resume()` 후 `wait_for_process()`로 최대 10초 폴링.
- 시나리오 도중 크래시 → 스텝 사이마다 `adb.is_running()` 확인, 죽으면 남은
  스텝을 건너뛰고 즉시 세션 정리(`crashed=True`로 표시, 부분 리포트는 생성).

**부가로 해결한 것**: A/B가 4주차 보고서에서 각각 남긴 "리포트가 실행마다
덮어써짐" 문제(발표용 스냅샷을 못 남기던 문제)를
`dynamic_report_{시나리오명}_{실행시각}.json` 형식으로 실행마다 별도 파일에
저장하도록 해결.

## 5일차 — 전원 통합 (미완료)

- ⬜ 실제 앱(정상 1개 + 악성 1개)으로 A→B→C→D 전체 파이프라인 end-to-end
  실행 — B와 같은 이유로 보류(anubis.apk 등 실제 악성 샘플 부재).
- ⬜ `login_flow`/`permission_request` 시나리오의 실제 좌표·권한 문자열 —
  실 대상 앱이 정해지면 jadx로 화면 리소스 ID 확인 후 채워야 함.
- ⬜ **`scenario_runner.py` 실제 실행 검증 안 됨**: 이 코드를 작성한
  환경(개발 컨테이너)엔 adb/frida-server/에뮬레이터가 전혀 없어서
  `python -m py_compile`로 문법 검사만 통과시켰음. A/B/C의 코드는 전부 각자
  실제 에뮬레이터(`xray_api30`, Pixel 4 / API 30)에서 검증했지만, D 코드는
  아직 그 환경에서 한 번도 실행해본 적이 없음 — **다음으로 할 일은 내
  로컬 개발 환경(에뮬레이터가 있는 Windows 머신)에서
  `feature/dynamic-agent` 브랜치를 pull받아 `scenario_runner.py`를 실제로
  돌려서 A/B/C 코드와 맞물리는지 확인하는 것.**

## 팀 논의 필요

- `controller._cleanup_current_session()`을 A의 private 메서드 그대로
  가져다 씀. D도 시나리오 실행마다(특히 반복 실행 시) 세션 정리가 필요해서
  어쩔 수 없이 재사용했는데, A 쪽에서 `cleanup()`처럼 public 메서드로
  노출해주면 더 깔끔할 것 같음.

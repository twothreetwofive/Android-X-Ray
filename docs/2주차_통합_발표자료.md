# 2주차 과제 통합본 (발표용)

> 팀원 4명(김은아, 백소정, 왕은서, 유예원)의 2주차 과제를 하나로 통합. 겹치는 내용은 정리하고, 내용이 갈리는 부분은 가장 완성도 높은 쪽을 채택함.

---

## 1. Android X-Ray란?

**한 줄 정의**: APK 파일 하나를 넣으면 정적분석 → 동적분석 → 네트워크분석을 자동으로 수행하고, 결과를 대시보드로 보여주는 원클릭 악성코드 분석 플랫폼.

지금까지 분석가가 수동으로 하나씩 하던 작업들 —
- JADX로 디컴파일
- Frida로 후킹
- tcpdump로 패킷 캡처

— 을 하나의 파이프라인으로 묶어서 **"APK 업로드 → 결과 대시보드"** 로 끝나게 만드는 것이 목표.

단순 탐지/차단 위주의 백신(Anti-Virus)과 달리, 난독화·암호화된 로직을 해제해서 분석가가 악성코드의 내부 구조와 동작 원리를 직관적으로 파악하도록 돕는 것이 핵심 목적.

### 왜 필요한가 — 유사 사례
- **Duo Security 'X-Ray for Android'**: 기기 내 권한 상승(Privilege Escalation) 등 알려진 취약점의 패치 여부를 스캔하는 진단 도구
- **KISA(한국인터넷진흥원) 'X-ray' 시스템**: 스미싱 문자 내 URL을 분석해 악성 앱 설치·피싱 사이트를 사전 차단하는 인프라
- **보안 학계 연구**: 앱을 샌드박스에서 구동시켜 런타임 행위를 추출·분석하는 동적 분석 자동화가 꾸준히 연구되는 분야

→ 이런 선례들처럼, 이번 프로젝트는 팀이 실제로 분석했던 **Anubis(안드로이드 뱅킹 악성코드 계열)** 사례를 기반으로 정적·동적·네트워크 분석을 자동화하는 것이 목표.

## 2. 핵심 모듈 구성 (4단계)

### ① 정적 분석 (Static Analysis)
앱을 실행하지 않고 코드/설정만 보고 분석.

- **디컴파일**: APK는 ZIP 파일이며 내부에 `classes.dex`(달빅 바이트코드)가 있음. 사람이 읽을 수 있게 변환하는 두 가지 도구:
  - **JADX**: DEX → Java 소스코드에 가깝게 복원. 코드 로직을 읽기 좋음.
  - **Apktool**: DEX → Smali(어셈블리 유사 중간언어) + 리소스 파일 복원. Manifest·리소스를 정확히 보거나 재조립이 필요할 때 사용.
  - Python에서는 별도 라이브러리 바인딩 없이 `subprocess.run(["jadx", "-d", output_dir, apk_path])` 형태로 CLI를 그대로 호출하면 됨.
- **AndroidManifest.xml 파싱**: `xml.etree.ElementTree`로 파싱. 위험 권한은 미리 딕셔너리로 정의해두고 매칭.
  - `<uses-permission>` 태그 기준 위험 권한 목록:
    | 권한 | 위험도 | 악용 예시 |
    |---|---|---|
    | `BIND_ACCESSIBILITY_SERVICE` | 고위험 | 화면 내용 훔쳐보기, 자동 클릭 |
    | `SYSTEM_ALERT_WINDOW` | 고위험 | 다른 앱 위 오버레이로 피싱 화면 덮어씌우기 |
    | `READ_SMS` / `RECEIVE_SMS` | 고위험 | 스미싱, 문자 탈취 |
    | `REQUEST_INSTALL_PACKAGES` | 중위험 | 추가 악성 APK 설치 |
  - `<intent-filter>`로 각 컴포넌트가 어떤 이벤트에 반응하는지도 함께 추출
- **위험도 점수화**: 권한별 가중치 합산 방식으로 1차 목표 설정 (여유 있으면 조합 패턴 가중치 추가)
- **출력**: `static_report.json` (권한 목록, 위험 권한 하이라이트, 점수, 컴포넌트 목록)

### ② 동적 분석 (Frida) — 리스크가 가장 큰 구간
통제된 에뮬레이터 환경에서 앱을 실제로 구동하며 런타임 행위를 관찰.

- **흐름**:
  1. 에뮬레이터에 `frida-server` 바이너리를 push하고 실행 (에뮬레이터 아키텍처 x86_64/arm64에 맞는 바이너리 필요)
  2. PC의 Python에서 `frida.get_usb_device()` → `device.spawn(package_name)` → `resume()`으로 앱 실행 제어
  3. Hook 스크립트(JS)를 `session.create_script(js_code)`로 주입 → `script.load()`
- **후킹 대상**: 악성코드는 탈취한 데이터를 평문으로 바로 보내지 않고, 보통 `StringBuilder.append`로 문자열을 조립 → `Base64.encode` 또는 XOR 같은 대칭 연산으로 난독화 → 서버로 전송. 이 함수들의 입력값을 후킹 시점에 가로채면 암호화 전 원본 데이터를 볼 수 있음.
  - `StringBuilder.append` (노이즈가 많아 최소 길이 필터 필요)
  - `android.util.Base64.decode/encode`
  - `javax.crypto.Cipher.doFinal` (표준 암호화 API는 이걸로 커버, **커스텀 XOR은 표준 API 후킹으로 못 잡으므로 문자열 조립 함수 후킹으로 우회 추적** 필요)
- **JS ↔ Python 통신**: `script.on('message', callback)` / JS 쪽 `send()`로 채널 구성
- **데이터 정제**: 중복 제거, 최소 길이 필터, Base64/URL 패턴 매칭으로 의미 있는 문자열만 추출
- **안전장치**: 앱 실행 관찰 시간 제한(예: 60~90초) + 크래시 시 재시작, 타임아웃 처리
- **출력**: `dynamic_report.json` (수집 문자열, 호출 함수명, 타임스탬프)

⚠️ **리스크**: 실제 악성코드는 에뮬레이터 탐지(anti-emulation) 로직이나 `frida-server` 감지 로직을 가진 경우가 많아, 에뮬레이터에서 아예 정상 동작하지 않을 수 있음. 8주차에 실제 샘플로 테스트할 계획이라면, **3~4주차 시점에 미리 1~2개 샘플로 이 문제를 검증**해두는 것이 안전함 (8주차에 발견하면 늦음).

### ③ 네트워크 분석 (Network Analysis)
앱과 외부 C&C 서버 간 통신을 캡처·분석.

- **캡처**: `tcpdump -i any -w capture.pcap`을 백그라운드 프로세스(`subprocess.Popen`)로 실행, 분석 종료 시 `.terminate()`. 동적 분석(Frida) 앱 실행 구간과 캡처 시작/종료를 동기화해야 함.
- **파싱 대상**:
  - DNS 쿼리 — 앱이 조회한 도메인 (C&C는 보통 도메인 기반 접속)
  - TLS ClientHello의 **SNI(Server Name Indication)** 필드 — HTTPS라 페이로드는 못 봐도, 접속하려는 도메인명은 평문으로 남음 (C&C 식별 핵심)
- **파싱 도구**: `scapy`로 직접 `DNSQR`/`TLS` 레이어를 순회할 수도 있지만, `tshark -r capture.pcap -T json`으로 이미 파싱된 JSON을 받는 편이 훨씬 편함. 실무에서도 tshark 1차 파싱 → Python 후처리 조합을 많이 사용.
- **C&C 판별 로직**: 화이트리스트(구글/광고 SDK 등 흔한 정상 트래픽)에 없는 도메인 + 하드코딩된 것처럼 보이는 IP를 걸러내는 간단한 규칙으로 충분.
- **중요한 설계 포인트**: HTTPS 트래픽은 SNI 정도만 보이고 실제 페이로드는 안 보임 → **이 한계는 Frida 쪽 메모리 후킹(평문 추출)으로 보완**하는 구조. 정적/동적/네트워크 세 모듈이 서로 다른 관점에서 상호보완적으로 설계된 것.
- **출력**: `network_report.json` (DNS 쿼리 목록, TLS SNI 목록, 의심 IP/도메인)

### ④ 통합 대시보드
- **Streamlit**이 이 프로젝트에는 최적. Flask보다 훨씬 적은 코드로 UI 구성 가능.
- **구조**: 사이드바(APK 업로드) + 메인 화면(정적/동적/네트워크/종합 점수 탭). 업로드 → 백그라운드에서 오케스트레이터 실행 → 진행 상황 표시 → 완료 시 결과 자동 렌더링.
- **시각화 요소**: 권한별 위험도 막대그래프, 추출 문자열 표, 의심 도메인 목록, 종합 위험도 점수 게이지, 위험도 색상 코딩(고/중/저), 결과 JSON/리포트 다운로드 버튼
- **예시 코드**:
  ```python
  import streamlit as st
  st.title("X-Ray 분석 결과")
  st.metric("위험도 점수", risk_score)
  st.subheader("위험 권한")
  st.table(dangerous_permissions)
  st.subheader("탈취된 평문 데이터 (Frida 후킹)")
  st.write(leaked_strings)
  st.subheader("C&C 후보 서버")
  st.table(cnc_candidates)
  ```

## 3. 시스템 아키텍처

```
                APK 업로드 (Web Dashboard)
                          │
        Python 메인 파이프라인 (Orchestrator)
                          │
        ┌─────────────────┼─────────────────┐
        ▼                 ▼                 ▼
 ① 정적 분석        ② 동적 분석         ③ 네트워크 분석
 (Static Analyzer)  (Dynamic Agent)     (Traffic Analyzer)
 JADX/Apktool 디컴   Frida 후킹,        tcpdump 캡처,
 파일, Manifest 파싱, 평문 데이터 수집    scapy/tshark 파싱,
 위험도 점수 산출                        C&C 서버 식별
        │                 │                 │
 static_report.json  dynamic_report.json  network_report.json
        └─────────────────┼─────────────────┘
                          ▼
              ④ 결과 통합 (JSON 취합)
           정적 + 동적 + 네트워크 → 종합 위험도 점수
                          │
                          ▼
              ⑤ 웹 대시보드 (Streamlit)
```

- **실행 순서에 대한 설계 포인트**: 정적 분석은 앱 실행이 필요 없어 독립적으로 먼저(또는 병행) 실행 가능. 반면 **동적 분석과 네트워크 캡처는 같은 앱 실행 구간 동안 동시에 진행**되어야 함 — 네트워크 캡처가 Frida로 앱을 실행하는 시간과 맞물려 있기 때문. 즉 "3개 모듈이 완전히 독립적으로 병렬 실행"이 아니라, **정적 분석(독립) → 동적+네트워크(같은 실행 구간에서 동시 진행)** 구조로 이해하는 것이 실제 구현과 더 맞음.
- **모듈 간 데이터 포맷(JSON 스키마)은 1주차에 미리 합의**해야 함 — 각 모듈이 서로 다른 형식으로 결과를 내면 6주차 통합 단계에서 꼬임.

## 4. 8주 개발 일정

| 주차 | 할 일 | 발표 내용 (격주) |
|---|---|---|
| 1주차 | 개발환경 구축 (adb, jadx-cli, apktool, Android Emulator, GitHub) + 아키텍처/JSON 스키마 확정 | — |
| 2주차 | 정적 분석 모듈 구현 (디컴파일 wrapper, Manifest 파싱, 위험도 점수) | **1차 발표**: 기획안 + 구조도 + 정적분석 스크립트 데모 |
| 3주차 | Frida 서버 연동, 앱 자동 설치/실행 제어 스크립트. 에뮬레이터 anti-emulation 회피 가능성 사전 검증 | — |
| 4주차 | 후킹 JS 스크립트 작성(StringBuilder/Base64/Cipher) + Python 에이전트 연동, 데이터 정제 로직 | **2차 발표**: Frida 연동 과정 + 앱 자동 실행 데모 + 평문 추출 결과 |
| 5주차 | 네트워크 캡처(tcpdump) + PCAP 파싱(DNS/SNI 추출), 화이트리스트 구축 | — |
| 6주차 | 정적→동적→네트워크 파이프라인 통합 (`main.py` 오케스트레이터), 종합 위험도 점수 로직 확정 | **3차 발표**: 네트워크 파싱 결과 + 통합 파이프라인 시연 |
| 7주차 | Streamlit 대시보드 UI 구축 (업로드 → 진행상황 → 결과 시각화) | — |
| 8주차 | **격리 네트워크 환경에서** Anubis 등 실제 뱅킹 악성코드 샘플로 엔드투엔드 테스트, 버그 수정, 수동 분석 대비 시간 단축 정량화 | **4차 발표**: 최종 데모 + 분석시간 단축 정량 결과 + 향후 개선 방향 |

### 개발 환경 / 도구
- `adb` — PC에서 에뮬레이터/기기를 제어 (앱 설치, 로그 확인 등)
- `jadx-cli`, `apktool` — 정적 분석용 디컴파일 도구
- Android Emulator — **Google APIs 이미지(루팅 가능)로 생성**. Google Play 이미지는 시스템 파티션이 읽기 전용이라 3~4주차 frida-server 설치 시 막힘
- `frida`, `frida-tools` — 동적 분석용 (`pip install frida-tools frida`)
- GitHub — 코드 버전관리/협업

## 5. 리스크 및 유의사항 (발표 시 강조 포인트)

1. **에뮬레이터 탐지 회피**: 실제 악성코드는 anti-emulation / frida-server 감지 로직을 갖는 경우가 많음. 8주차에 가서 발견하면 늦으므로, 3~4주차에 미리 실제 샘플 1~2개로 검증 필요.
2. **HTTPS 트래픽의 한계**: 네트워크 분석은 SNI(목적지 도메인)까지만 보이고 페이로드 내용은 못 봄 → Frida 메모리 후킹이 이를 보완하는 상호보완적 설계임을 명확히 설명.
3. **실제 악성코드 테스트 시 네트워크 격리 필수**: 8주차에 Anubis 등 실제 샘플을 구동할 때, 진짜 C&C 서버로 아웃바운드 연결이 나가지 않도록 로컬 프록시/격리망 구성 + 에뮬레이터 스냅샷 백업이 반드시 선행되어야 함 (안전 이슈).
4. **모듈 간 JSON 스키마 사전 합의**: 6주차 파이프라인 통합 시 포맷 불일치로 꼬이지 않도록 1주차에 `static_report.json` / `dynamic_report.json` / `network_report.json`의 필드명·구조를 미리 확정.
